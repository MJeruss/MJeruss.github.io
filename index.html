Baseball Simulator

The primary motivation behind this project was to use test driven development to create a fun tool that also demonstrates our understanding of the four pillars of object oriented programming. As baseball fans, we thought creating a robust baseball simulator would be a challenging but rewarding goal.

Our baseball simulator allows a user to select two teams from any of the Major League Baseball teams and simulate a baseball game. There are three options for a user: simulate a single at bat at a time, simulate the rest of an inning, or simulate the rest of the game. All of the options have different behavior for displaying the state of the game. At the end of each at bat, information about the at bat is displayed as well as information about the next at bat. At the end of an inning, an inning summary is printed and information about the next at bat is printed. At the end of the game, a game summary and box score are printed.

For our project, we decided to use Java. We chose java for a number of reasons. The first reason we chose java is because it is what we have been using this semester in our class. The second reason is that java is a strictly typed langauge and can be verbose--this means it is important to have self-documenting code that uses good names for variables, functions, and classes. We thought that because we are simulating a baseball game and were going to have a lot of classes and functions that represent different attributes of a baseball game, it would make sense to go with java. When it came to the design, we decided that we were going to split things up into 14 classes.

The classes are as follows:

AT_BAT_RESULT
Batter
Game
GameSimulator
GameStats
Inning
Pitcher
Player
PlayGame
Scoreboard
SetupGame
SetupGameOutput
Team
UserInput

GameSimulator is the main driver for the whole program--it sets up a game and plays the game. The setup of the game is handled by SetupGame, SetupGameOutput and UserInput. The PlayGame class plays the game and also uses UserInput. The game class has two Team objects, an Inning Object which tracks the state of the innings in a game, and a GameStats object--all of the methods from these classes are wrapped by Game.java. The Team objects have Player objects. The GameStats object keeps track of the game. The Game class calls static methods from the Scoreboard class which is responsibile for printing out updates about what is going on in the game. In addition to all of the classes, we also had test classes for all of the above classes as we took a test driven development approach to this project.

When it came to implementation, my partner Cory, and I used github to collaborate. While we did do some paired programming and combine our brain power for the initial design and for other bits and pieces throughout the project, we tried our best to plan and delegate work so that we could implement individually.

As I mentioned above, my partner and I took a test driven development approach meaning that we chose to look before we leaped. We used PlantUML to design our entire project and wrote all of the tests for every individual method in the entire project before we implemented the methods. The truth is, at times, this was a frustratring process that felt slow to get moving but paid off in the end. Because of our meticulous planning and design, when we put everything together at the end, it all came together quite seemlessly--we were both impressed. Even though what the project accomplishes is rather simple, we tried to write robust code and I believe the methodologies we used and skills we practiced while collaborating were very valuable. I am grateful to have had this opportunity and partner to practice what I've been taught!
